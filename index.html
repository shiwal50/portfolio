The hypothesis
Enterprise clients were abandoning transactional workflows not because of user error, but because the platform had no native reliability layer — forcing dangerous workarounds that silently failed at scale.
How I found it
Noticed 3–5 enterprise clients per week reporting failed transactional sends, each escalated as a one-off. Ran my own analysis across ticket history — found the same three failure modes every time: no delivery status visibility, no retry logic, no failure alerting. Multiple accounts had built manual workarounds (spreadsheet-tracked resend queues), indicating the problem was critical enough to absorb significant operational cost.
Trade-offs considered (MECE)
Option A
Documentation fix only — rejected. The failure was architectural, not instructional.
Rejected
Option B
Per-client workaround support — rejected. Does not scale, creates tech debt.
Rejected
Option C
Native transactional layer (JetSend) — addresses root cause, scales, closes competitive gap.
Selected ✓

Honest reflection: I did not own a formal post-launch metrics review. I know ticket volume dropped from qualitative signal and team feedback — but I did not establish a rigorous before/after baseline. In future work, I will define success metrics upfront, agreed with PM before launch, not after.

The problem
5 white-label platform owners on HighLevel flagged they could not handle end-client ticket volume manually. Each was routing every inbound query to a human agent, creating a bottleneck that was degrading CSAT across their client base.
What I did
Proposed an AI agent embedded in the support flow — able to handle FAQs, qualify issues, and escalate only what required human judgment. Aligned product, engineering, and design on scope, wrote the brief, coordinated UAT, and tracked post-launch defect closure. Shipped in 6 weeks.
